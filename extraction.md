# AI Extraction Strategy

## Overview

The objective of this project is to transform a collection of Environmental Product Declaration (EPD) PDF documents into structured JSON that can be consumed directly by the comparison application in Part 2.

Rather than extracting plain text, the goal was to produce a deterministic and validated data model while preserving traceability back to the original document.

The resulting JSON becomes the application's canonical data source.

---

# Overall Strategy

The extraction pipeline follows a structured workflow rather than relying solely on a single AI response.

```
EPD PDF

↓

Google Files API

↓

Gemini 2.5 Flash

↓

Structured JSON

↓

Response Schema

↓

Zod Validation

↓

Application Metadata Injection

↓

JSON Output
```

Each stage has a single responsibility.

* Gemini extracts information from the document.
* Response Schema constrains the response format.
* Zod validates the generated JSON.
* The application injects deterministic metadata such as document IDs and extraction timestamps.

This separation keeps the AI responsible only for information extraction while allowing the application to control consistency.

---

# Model and Architecture

I selected **Gemini 2.5 Flash** together with Google's Files API.

Two approaches were considered:

1. Extract PDF text first, then send the text to the model.
2. Upload the original PDF and allow the multimodal model to interpret the document directly.

The second approach was selected.

Environmental Product Declarations contain complex tables, multiple layouts, footnotes, and multi-column formatting. Converting these documents into plain text would lose structural information that is important for correctly identifying lifecycle stages and environmental indicators.

Uploading the original PDF preserves document context and significantly improves extraction accuracy.

After the PDF is uploaded, Gemini generates structured JSON constrained by a Response Schema before the output is validated using Zod.

---

# Accuracy

Several techniques were combined to improve extraction quality.

## Response Schema

Gemini is configured with a Response Schema so the model is guided toward the expected JSON structure instead of inventing its own format.

## Schema Validation

Every response is validated using Zod.

If required fields are missing or incorrect data types are returned, extraction immediately fails instead of producing inconsistent data.

## Provenance

Every extracted environmental value contains provenance information:

* page
* section
* table
* rawText

This allows every reported value to be traced back to the original location inside the EPD.

## Deterministic Model Configuration

The model is executed with a temperature of **0** to reduce output variability.

## Automatic Retry

Recoverable API failures such as **429** and **503** are automatically retried using exponential backoff.

# Research and Process

The extraction pipeline evolved through several iterations.

The initial implementation relied primarily on prompt engineering. Although the model produced useful results, the JSON structure occasionally varied between documents.

To improve consistency, I introduced Gemini's Response Schema together with Zod validation. This significantly reduced malformed responses and ensured every accepted extraction matched the expected schema.

Another improvement was moving technical metadata generation out of the AI model.

Fields such as document identifiers and extraction timestamps are generated by the application after validation rather than by the language model. This guarantees deterministic metadata across every extracted document.

During batch testing another practical issue became apparent.

Some Environmental Product Declarations are substantially larger than others. Attempting to extract every environmental indicator together with provenance for every lifecycle stage produced responses large enough to approach the model's output limit, resulting in truncated JSON.

Rather than introducing a significantly more complex multi-pass extraction pipeline, I revisited the assessment requirements.

The application in Part 2 primarily compares products using embodied carbon across lifecycle stages. Therefore, the extraction schema was intentionally refined to capture only the environmental information required by the application instead of every available indicator within the EPD.

This reduced response size considerably while remaining fully aligned with the assessment objectives.

# Batch Extraction Strategy

The extractor processes documents sequentially.

```
EPD 1

↓

Extract

↓

Save JSON

↓

EPD 2

↓

Extract

↓

Save JSON
```

Although parallel extraction using `Promise.all()` would reduce execution time, I intentionally chose sequential processing.

There are two reasons for this decision.

First, Gemini API requests are subject to rate limits and temporary service availability. Sending many large PDF requests simultaneously significantly increases the probability of receiving HTTP 429 (Too Many Requests) or HTTP 503 (Service Unavailable) responses.

Second, sequential processing provides much better fault isolation. If one document fails validation or extraction, the remaining documents continue processing normally, making it much easier to identify and debug problematic EPDs.

Since dataset generation is typically an offline preprocessing task rather than a user-facing request, reliability was prioritised over raw throughput.

# Logging

Every extraction generates debugging information.

```
logs/

├── gemini-response.json
└── gemini-error.txt
```

These logs allow failed extractions to be inspected without rerunning the model.

# Output

Each EPD produces one validated JSON document.

```
epds/

    Product.pdf

↓

data/

    Product.json
```

The JSON contains all information required by the comparison application while remaining deterministic and fully validated before being written to disk.

---

# Conclusion

The final solution prioritises deterministic structured data over raw AI output.

By combining Gemini's multimodal PDF understanding, Response Schema enforcement, Zod validation, provenance tracking, automatic retry, and application-generated metadata, the extraction pipeline produces a reliable dataset suitable for downstream comparison and visualisation.

Throughout development, engineering decisions were driven by the requirements of the final application rather than attempting to extract every possible value from the source documents. This resulted in a simpler, more reliable extraction pipeline while remaining fully aligned with the objectives of the assessment.
